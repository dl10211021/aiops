# Safety Policy Action Design

## 背景

当前安全策略页面直接暴露后端内部字段，例如审批正则、只读拦截正则和硬拦截片段。这个模型对开发者可用，但对运维管理员不友好：基础用户难以判断应该把命令放到审批、只读拦截还是硬拦截里，也不容易理解 HTTP API、虚拟化、Kubernetes、S3 等平台动作的真实风险。

整改目标是把安全策略从“正则规则编辑器”升级成“动作策略配置”。用户用业务语言配置“哪个资源域、哪个平台、哪个对象、哪个动作、如何处理”，系统再把它转换到底层命令、正则、HTTP 方法、API 路径或请求体匹配。

## 设计原则

1. 只读模式不作为用户需要维护的一类规则展示。它是系统行为：只允许查看、巡检、诊断。
2. 用户主要配置两类风险动作：需要审批、禁止执行。允许执行用于明确放行常见只读动作和低风险动作。
3. 普通模式不要求用户懂正则。正则、API Path、JSON 条件放到高级模式。
4. 数据中心平台不能只按协议分类。HTTP API 只是传输方式，VMware 删除虚拟机和 Prometheus 查询指标不能共用同一组安全语义。
5. 策略必须支持范围控制，例如资产组、标签、环境、平台、操作来源。

## 用户可见模型

安全策略页面展示三种处理方式：

- 允许执行：查看状态、读取日志、查询配置、指标查询等低风险动作。
- 需要审批：可能改变系统状态，但可以在人工确认后执行的动作。
- 禁止执行：无论只读还是读写模式都拒绝执行的动作。

只读会话和读写会话的解释如下：

| 策略动作 | 只读会话 | 读写会话 |
| --- | --- | --- |
| 允许执行 | 允许 | 允许 |
| 需要审批 | 拦截并提示切换读写或发起审批 | 进入审批 |
| 禁止执行 | 禁止 | 禁止 |

## 规则作用范围

规则新增或编辑时使用以下字段：

- 资源域：主机系统、数据库、中间件、容器与云原生、虚拟化与私有云、网络与安全设备、存储与备份、监控与告警、硬件带外管理、平台自身能力。
- 平台类型：Linux、Windows、MySQL、Redis、Kubernetes、VMware、Proxmox、OpenStack、ZStack、Prometheus、Grafana、S3、MinIO、Ceph RGW 等。
- 资源对象：服务、文件、数据库表、Pod、Deployment、Namespace、虚拟机、宿主机、Bucket、Object、Bucket Policy、告警规则等。
- 动作类型：查看、创建、修改、删除、重启、停止、扩容、缩容、迁移、快照、回滚、清空、格式化、权限变更、配置下发。
- 生效资产：全部、指定资产类型、指定资产组、指定标签、指定环境、指定单资产。
- 操作来源：聊天、巡检、定时任务、告警联动、子任务、API。
- 处理方式：允许执行、需要审批、禁止执行。
- 规则说明：面向管理员的人类可读说明。
- 示例：展示命令、SQL、API Path 或平台动作示例。

## 资源域建议

### 主机系统

- Linux / Unix
- Windows Server
- KVM / Libvirt Host

典型动作：

- 查看状态：允许执行
- 查看日志：允许执行
- 重启服务：需要审批
- 修改配置文件：需要审批
- 删除文件：需要审批
- 格式化磁盘：禁止执行
- 关机 / 重启主机：需要审批或禁止执行，由环境策略决定

### 数据库

- MySQL
- PostgreSQL
- Oracle
- SQL Server
- MongoDB
- Redis
- Elasticsearch

典型动作：

- 查询数据结构和状态：允许执行
- 查询少量业务数据：可配置，默认需要审批
- INSERT / UPDATE / DELETE：需要审批
- DROP / TRUNCATE：禁止执行
- FLUSHALL / FLUSHDB：禁止执行

### 容器与云原生

- Docker
- containerd
- Kubernetes
- Harbor

典型动作：

- 查看 Pod、Node、Deployment：允许执行
- 查看事件和日志：允许执行
- Deployment 扩缩容：需要审批
- 重启 Workload：需要审批
- 删除 Pod：需要审批
- 删除 Namespace：禁止执行
- 修改 RBAC / Secret：需要审批或禁止执行

### 虚拟化与私有云

- VMware vCenter / ESXi
- Proxmox VE
- OpenStack
- ZStack
- Hyper-V
- KVM / Libvirt

典型动作：

- 查看虚拟机状态：允许执行
- 查看宿主机和资源池：允许执行
- 创建快照：需要审批
- 回滚快照：需要审批
- 重启虚拟机：需要审批
- 迁移虚拟机：需要审批
- 删除虚拟机：禁止执行
- 删除存储卷：禁止执行

### 网络与安全设备

- Switch / Router
- Firewall
- F5 BIG-IP
- A10 Load Balancer
- WAF
- VPN Gateway
- DNS Server

典型动作：

- show / display / ping / traceroute：允许执行
- 进入配置模式：需要审批
- 保存配置：需要审批
- 下发 ACL / NAT / 路由变更：需要审批
- 清空配置 / reset saved-configuration：禁止执行
- 重启核心网络设备：需要审批或禁止执行

### 存储与备份

- Ceph
- NFS
- NAS / SAN
- MinIO
- S3 / S3 Compatible
- AWS S3
- 阿里云 OSS
- 腾讯云 COS
- 华为 OBS
- Ceph RGW
- 备份平台

S3 对象存储需要独立处理，不能简单等同于 HTTP API。

典型动作：

- 查看 Bucket 列表：允许执行
- 查看对象列表：允许执行
- 查看对象元数据：允许执行
- 下载对象：可配置，默认需要审批
- 批量下载对象：需要审批或禁止执行
- 上传对象：需要审批
- 修改对象标签 / ACL / 生命周期：需要审批
- 修改 Bucket Policy：需要审批
- 公开 Bucket：禁止执行
- 删除对象：需要审批或禁止执行
- 批量删除对象：禁止执行
- 清空 Bucket：禁止执行
- 删除 Bucket：禁止执行
- 访问密钥 / 凭据操作：需要审批或禁止执行

典型动作：

- 修改告警规则：需要审批

### 硬件带外管理

- 回滚技能：需要审批

- 修改安全策略：需要审批或管理员权限

## 普通规则配置

普通用户通过表单新增规则：

- 规则名称：生产环境禁止删除虚拟机
- 资源域：虚拟化与私有云
- 平台类型：全部
- 资源对象：虚拟机
- 动作类型：删除
- 生效资产：标签包含 生产
- 操作来源：全部
- 处理方式：禁止执行
- 说明：生产虚拟机删除必须由人工平台操作，不允许 AI 执行。

另一个例子：

- 规则名称：Linux 重启服务需要审批
- 资源域：主机系统
- 平台类型：Linux
- 资源对象：服务
- 动作类型：重启
- 生效资产：全部
- 操作来源：聊天、定时任务、告警联动
- 处理方式：需要审批
- 示例：systemctl restart nginx

## 命令规则配置

主机、网络设备、本地脚本类动作仍然需要支持直接新增命令规则。

匹配方式：

- 命令开头：例如 `systemctl restart`
- 包含关键词：例如 `mkfs`
- 完全等于：例如 `reboot`
- 参数包含：例如 `--force`
- 正则匹配：高级模式

处理方式仍然是允许执行、需要审批、禁止执行。

## API / 平台动作规则配置

虚拟化、Kubernetes、监控、存储、S3 等平台类能力使用平台动作规则。

普通模式字段：

- 平台类型
- 资源对象
- 动作类型
- 处理方式

高级模式字段：

- HTTP 方法
- API Path
- 请求体字段
- JSON 条件
- 正则

示例：

- VMware / 虚拟机 / 删除：禁止执行
- VMware / 虚拟机 / 重启：需要审批
- Kubernetes / Deployment / 扩缩容：需要审批
- Kubernetes / Namespace / 删除：禁止执行
- Prometheus / 指标查询：允许执行
- S3 / Object / 下载：需要审批
- S3 / Bucket / 删除：禁止执行

## 前端整改建议

安全策略页面从当前的文本域编辑改为：

1. 策略总览：展示当前允许、审批、禁止规则数量和最近命中记录。
2. 资源域侧栏：按数据中心资源域分类。
3. 动作策略表：展示规则名称、平台、资源对象、动作、范围、处理方式、启用状态。
4. 新增规则向导：基础用户通过中文表单创建规则。
5. 高级规则折叠区：显示正则、命令匹配、API Path、JSON 条件。
6. 规则测试器：输入一条命令或 API 动作，页面显示会被允许、审批还是禁止。

## 后端整改建议

短期不需要推翻现有 `safety_policy.json`。可以先新增语义化规则层：

```json
{
  "rules": [
    {
      "id": "linux-service-restart-approval",
      "name": "Linux 重启服务需要审批",
      "domain": "os",
      "platform": "linux",
      "resource": "service",
      "action": "restart",
      "scope": {"tags": []},
      "sources": ["chat", "cron", "alert"],
      "decision": "approval",
      "matchers": [
        {"type": "command_prefix", "value": "systemctl restart"},
        {"type": "regex", "value": "\\bservice\\s+\\S+\\s+restart\\b"}
      ],
      "description": "重启服务会影响业务可用性，必须人工确认。"
    }
  ]
}
```

然后兼容现有字段：

- 新 UI 写入语义化规则。
- 后端执行时同时支持语义化规则和旧字段。
- 旧字段只在高级模式中展示。
- 默认策略从旧结构迁移到新结构，但不删除旧兼容逻辑。

## 迁移路线

第一阶段：前端语义化展示，不大改后端。

- 将现有规则映射为中文动作表。
- 隐藏只读拦截文本域。
- 增加允许 / 审批 / 禁止三态说明。
- 增加高级模式，保留原始规则编辑能力。

第二阶段：新增语义化规则结构。

- 后端新增 `rules`。
- 增加 matcher 类型：command_prefix、contains、equals、regex、http_method、api_path、json_field、platform_action。
- 增加规则测试接口。
- 保留旧字段兼容。

第三阶段：策略治理能力。

- 规则命中审计。
- 规则版本记录。
- 规则导入 / 导出。
- 策略变更审批。
- 生产环境模板、测试环境模板、严格模板。

## 验收标准

- 基础用户不用正则也能新增审批和禁止规则。
- 安全策略页能清楚解释只读、读写、审批、禁止执行的关系。
- 虚拟化、Kubernetes、S3、监控、存储等平台不再只显示为 HTTP API。
- 用户可以为生产标签、指定资产组、指定操作来源配置不同策略。
- 规则测试器能解释一条命令或平台动作为什么被允许、审批或禁止。
- 旧 `safety_policy.json` 继续可用，不影响现有审批和硬拦截能力。
